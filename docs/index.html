基于 rapier3d-f64 的 Rust 原生物理运行时。为 Java Minecraft (NeoForge) 提供 刚体动力学、空间查询、体素碰撞、航天工程、混沌理论等 590 个 C ABI 函数。 Kahan 补偿求和 + mul_add 融合乘加确保双精度稳定性，无锁 Ring Buffer + 零拷贝 Bridge 实现零帧事件开销。

Interaction — 刚体交互力系统

刚体间物理交互在 world_step 内自动执行。初始化时通过 world_set_*_law 配置一次，之后每帧零额外 FFI 调用。

三大交互力

系统物理模型配置
万有引力F = G·m₁·m₂/r² (O(n²) 直接求和)NewtonGravityLaw — 支持自定义 G、最小/最大距离
库仑摩擦力F_f = μ·F_n，静/动摩擦自动切换CoulombFrictionLaw — 读取 narrow-phase 接触数据施加
空气阻力Stokes (低 Re) + 湍流 (高 Re) 双模式AirDragLaw — 雷诺数自适应切换

Java 使用示例

// === 初始化（仅一次）===
PhysicsWorld world = new PhysicsWorld(0, -9.81, 0);

// 1. 启用刚体间万有引力 (G=1000 适合游戏尺度)
world.setNewtonGravityLaw(new NewtonGravityLaw()
    .gravitationalConstant(1000.0)
    .minDistance(0.01)
    .enabled(true));

// 2. 启用库仑摩擦力
world.setCoulombFrictionLaw(new CoulombFrictionLaw()
    .staticCoefficient(0.6)
    .dynamicCoefficient(0.4)
    .velocityThreshold(0.01)
    .enabled(true));

// 3. 启用空气阻力
world.setAirDragLaw(new AirDragLaw()
    .fluidVelocity(0, 0, 0)
    .density(1.225)         // 空气密度 kg/m³
    .dragCoefficient(0.47)  // 球体阻力系数
    .referenceArea(0.5)     // 参考面积 m²
    .enabled(true));

// === 每帧循环 ===
while (running) {
    world.step();  // 三种交互力自动执行
    // 无需额外调用
}

Bridge — 零拷贝 Java-Native 桥接层

传统 JNI 每帧为每个刚体分配 newDoubleArray(3) 拷贝位置/速度。 零拷贝桥接层通过 DirectByteBuffer 共享内存和预分配槽位消除全部 per-frame 分配。

Java 使用示例

// 获取 Unsafe 实例 (所有 JVM 通用，含 Java 21+)
private static final Unsafe U = getUnsafe();
private static Unsafe getUnsafe() {
    try {
        Field f = Unsafe.class.getDeclaredField("theUnsafe");
        f.setAccessible(true);
        return (Unsafe) f.get(null);
    } catch (Exception e) { throw new Error(e); }
}

// === 初始化：分配本地内存，全帧复用 ===
long bufPtr = U.allocateMemory(MAX_BODIES * 13 * 8);
U.setMemory(bufPtr, MAX_BODIES * 13 * 8, (byte) 0);

// === 每帧：一次调用获取全部刚体状态 ===
int count = RigidBodyNative.bridgeBulkBodySnapshot(world, bufPtr, MAX_BODIES);
for (int i = 0; i < count; i++) {
    long off = bufPtr + i * 13 * 8;
    double tx = U.getDouble(off);        // translation.x
    double ty = U.getDouble(off + 8);    // translation.y
    double tz = U.getDouble(off + 16);   // translation.z
    double vx = U.getDouble(off + 56);   // linvel.x
    // ... 同步到 Minecraft Entity ...
}

// === 预分配槽位 (代替 new double[3] 模式) ===
long posSlot = U.allocateMemory(24);
RigidBodyNative.bridgeVec3ToSlot(x, y, z, posSlot);
double x = U.getDouble(posSlot);
double y = U.getDouble(posSlot + 8);
double z = U.getDouble(posSlot + 16);

Minecraft 性能优化指南

1. 本地库加载

  • 库在首次加载时从 JAR 提取到临时目录
  • 后续加载使用缓存路径
  • 建议在 Mod 构造函数中预加载: System.loadLibrary("mps_rigid_body")

2. 线程安全

  • Rust 侧使用 parking_lot::Mutex 保护事件队列
  • 物理步进在每个 World 内单线程
  • 多个 World 可并行运行 (rayon 线程池)

3. 内存管理

  • 不透明句柄必须显式销毁 (无 GC 自动回收)
  • JNI: long 值作为原始字段存储
  • FFM: MemorySegment 通过 Arena 作用域管理
  • 使用 try-with-resources: try (PhysicsWorld world = ...) { ... }

4. 区块化物理

  • 为每个加载的区块区域创建独立的 Rapier World
  • 在区块边界同步碰撞信息
  • 卸载区块时销毁对应的 World

5. 体素碰撞体缓存

  • 体素碰撞体生成开销大 — 缓存每个区块的碰撞体
  • 仅在方块变更时重新生成
  • 默认阈值: small_voxel_limit=128, mesh_voxel_limit=20_000

6. 空间查询优化

  • 使用 CRbTree/RTree 进行实体查找，避免遍历所有实体
  • 每 tick 插入实体 AABB，按区域查询
  • 批量查询减少 Java/native 调用开销

7. 事件处理

  • 每 tick 后批量读取事件 (一次 native 调用)
  • 及时清理事件队列: world.clearEvents()
  • 最大事件数: 16,384

事件缓存注册机制 — 初始化注册，零帧开销

传统的事件读取模式每帧需要：count → allocate → read → parse 四次 FFI 调用。 缓存注册机制将事件分发路径压缩为一次 drain 调用，或直接通过 C 函数指针回调。

核心组件

1. 无锁 Ring Buffer

单生产者（Rust 物理线程）+ 单消费者（Java drain 线程）的无锁环形缓冲区。 使用 AtomicU32 write/read 游标实现 SPSC 队列语义，零内存分配： 

// Rust 侧 — 物理线程写入
fn push(&self, event: CollisionEventRecord) {
    let w = self.write.load(Ordering::Relaxed);
    let r = self.read.load(Ordering::Acquire);
    if w.wrapping_sub(r) >= cap {
        self.dropped.fetch_add(1, Ordering::Relaxed); // 溢出统计
        return;
    }
    unsafe { (*self.buf.get())[(w % cap) as usize] = event; }
    self.write.store(w.wrapping_add(1), Ordering::Release);
}

2. Callback Registry

初始化时注册 C 函数指针，物理步进时直接调用，无需 Java 侧每帧轮询： 

// Rust 侧 — 回调槽位（零堆分配）
struct CallbackSlot {
    cb: usize,       // C 函数指针 (0 = 未设置)
    user_data: usize, // 透传用户数据
    handle: u64,     // 用于取消注册
}

fn dispatch_collision(&self, record: CollisionEventRecord) {
    if self.collision_cb.cb != 0 {
        let cb: CollisionEventCallback = unsafe { std::mem::transmute(self.collision_cb.cb) };
        if let Some(f) = cb {
            unsafe { f(ptr::null(), &record, self.collision_cb.user_data as *mut c_void); }
        }
    }
}

精度优化 — 双精度数值稳定性

项目全程使用 rapier3d-f64 双精度物理引擎，并在公式层实施了多层精度保障措施， 确保从纳米级量子效应到天文尺度轨道力学的高精度计算。

Kahan 补偿求和

当大量小值累加到已累积的大值时，普通 x += y 会丢失低位有效数字。 Kahan 算法通过追踪舍入误差补偿项，将累积精度从 ~15 位提升到 ~30 位有效数字。 

// math.rs — Kahan 补偿累加器
pub(crate) struct KahanSum {
    sum: f64,          // 当前和
    compensation: f64, // 被丢弃的低位误差
}
impl KahanSum {
    pub fn add(&mut self, value: f64) {
        let y = value - self.compensation;
        let t = self.sum + y;
        self.compensation = (t - self.sum) - y;  // 恢复丢失的低位
        self.sum = t;
    }
}

除标量 KahanSum 外，项目还提供了 KahanVec3 — 对三维向量分量独立 补偿求和。应用于气动面表面积分、SPH 粒子邻居遍历、软体约束求解等需要高精度向量累加的场景。
